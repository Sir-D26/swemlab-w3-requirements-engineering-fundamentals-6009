Week 3 Lab: Requirements Engineering for Campus Event Finder
Scenario

This lab focuses on defining the requirements for a proposed mobile application called Campus Event Finder. The goal of the application is to help university students easily discover and attend events happening on or around campus, such as club meetings, sports events, academic workshops, and guest lectures. The app will allow students to view event listings, filter events by category, and mark events they are interested in so they can receive reminders.

Key Takeaways

Functional Requirements (FRs) describe what the system should do, such as displaying events, filtering events by category, and allowing students to mark events as interested.

Non-Functional Requirements (NFRs) define the quality and performance of the system, such as usability, reliability, and response time.

Requirements elicitation involves gathering information from stakeholders (students, event organizers, and university staff) through discussions, interviews, or surveys.

Clear documentation in a Software Requirements Specification (SRS) helps developers understand the system and reduces misunderstandings during development.

Separating FRs and NFRs helps ensure both system functionality and system quality are properly defined.

Challenges

One challenge I faced was distinguishing between functional and non-functional requirements. Some requirements seemed similar at first. I addressed this by asking whether the requirement described a system action (functional) or a quality constraint (non-functional).
